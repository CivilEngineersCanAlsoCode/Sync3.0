**What exact problem were you solving?**  
“In Sprinklr Insights, the biggest problem was not lack of features — it was the setup friction. A customer could not get value quickly because the whole Insights workflow was consultant-driven and highly manual. To start using the product, teams had to manually create topics by entering lots of keywords, handles, hashtags, and brand terms, then manually create themes for sub-categories, and then manually build dashboards for each use case like brand monitoring, crisis monitoring, campaign monitoring, and so on. So the product was strong, but the path to reach value was slow, complex, and inconsistent across clients.”

**For whom specifically?**  
“I was solving this for two sets of people. First, the end users — analysts, managers, and leadership — who wanted insights fast but were blocked by complexity and heavy configuration. Second, the consultants and account teams who were doing repeated setup work for every new client. Analysts wanted deep reporting, managers wanted operational views and faster monitoring, and leadership wanted quick executive insights without navigating a heavy enterprise UI. On the internal side, consultants were spending weeks setting up similar things again and again, which created a scalability bottleneck.”

**What was the pain intensity (quantified)?**  
“The pain was very measurable. Onboarding a new Insights client typically took around **4+ weeks** because it depended on consultants to configure topics, themes, dashboards, and reporting. Also, building persona-based experiences was a manual exercise — roughly **5 hours per persona** to create and deploy. So time-to-insight was slow, time-to-value was slow, and a lot of effort was repeated across clients. The product was not failing because of capability — it was failing because it was too hard to adopt quickly.”

**What happens if this problem remains unsolved?**  
“If we don’t solve this, three things happen. One, customers take too long to see value, so adoption stays weak and churn risk increases — especially in the early phase when they’re deciding whether the product is worth it. Two, we stay dependent on consultants for basic setup, which means we can’t scale onboarding as fast as sales growth. Three, even after onboarding, leadership users keep disengaging because the UI and workflow feel overwhelming, so the product becomes ‘used by a few experts’ instead of being widely adopted across the organization.”

**Why was this urgent?**  
“It was urgent because onboarding speed directly affects revenue and retention in enterprise SaaS. When customers buy an Insights platform, they expect results quickly. If it takes weeks to configure basic reporting, the customer starts questioning the value, internal champions lose momentum, and renewals become harder later. Also internally, consultant bandwidth becomes a constraint — if onboarding takes 4 weeks and requires heavy effort, we either slow down new implementations or we increase services cost, both of which hurt the business.”

**Was this must-have or nice-to-have? Why?**  
“This was a must-have because it was a foundational bottleneck. Improving setup and time-to-insight is not a ‘nice UX improvement’ — it is what makes the product scalable and adoptable. Without solving this, adding more features doesn’t help because customers can’t reach the value of the existing features fast enough. My goal was to convert Insights from a consultant-led project into a self-serve product experience, where clients can go from entering a brand to getting useful dashboards and actions quickly.”
